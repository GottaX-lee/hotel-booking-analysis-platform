# 项目规格书：酒店预订需求智能数据分析与可视化平台

## 第一章：项目核心红线（AI 绝不可触碰的禁忌）

以下规则具有最高优先级，任何代码生成都不得违反。

| 序号 | 红线内容 | 原因（为什么人类禁止你这么做） | 惩罚性指令 |
|------|----------|------------------------------|------------|
| R-01 | 严禁保留 `reservation_status` 和 `reservation_status_date` 列 | 这两列是订单结束后的"未来信息"。若保留，模型会学到"因为最终取消了，所以预测取消"的荒谬逻辑，导致数据泄露。 | 必须在数据加载后第一行代码中永久物理删除这两列，后续任何环节（EDA、建模、预测）都不允许再引用。 |
| R-02 | 严禁使用 `reservation_status` 列进行任何可视化或相关性分析 | 即使是为了画图，只要引用该列，就等于让人类看到了"未来答案"，会污染分析思路。 | 如果在 EDA 阶段 AI 生成的代码中出现了这两列，视为严重违规。 |
| R-03 | 严禁将"泄露测试"的结果用于反向调参 | 最后会做一个"对照实验"（用含泄露列的旧数据测试模型），但这仅用于展示危害。绝对禁止根据这个高得离谱的准确率去调整模型参数。 | 对照实验仅输出对比表格，不参与模型迭代。 |
| R-04 | 严禁擅自决定核心业务特征的生成方式 | 例如 `total_stay` 和 `total_guests` 的计算逻辑、时间维度的聚合粒度（日/周/月），必须由人类确认。 | AI 只能按人类给出的公式写代码，不得发明新的合成特征。 |

## 第二章：数据集说明（基于用户实际数据）

### 基本信息

- **数据来源**：阿里天池平台（Hotel Booking Demand 数据集）
- **数据规模**：约 119,000+ 条记录，32 个字段
- **目标变量（标签 y）**：`is_canceled`（0 = 未取消，1 = 取消）—— 此列不删除，用于训练和预测
- **必须删除的列（特征 X）**：`reservation_status`、`reservation_status_date`（立即删除，物理丢弃）

### 关键字段速查表（供 AI 编码时对照）

| 字段名 | 业务含义 | AI 处理注意事项 |
|--------|----------|----------------|
| `hotel` | 酒店类型（Resort Hotel / City Hotel） | 类别特征，需编码 |
| `lead_time` | 提前预订天数 | 重要数值特征 |
| `arrival_date_month` | 到达月份（英文缩写，如 July） | 需映射为数字 1-12 |
| `stays_in_weekend_nights` | 周末入住夜数 | 参与合成特征 |
| `stays_in_week_nights` | 周中入住夜数 | 参与合成特征 |
| `adults` / `children` / `babies` | 各类人数 | 处理 NULL，参与合成特征 |
| `agent` | 旅行社 ID | 大量 NULL，填充为 0（表示无代理） |
| `adr` | 日均房价（Average Daily Rate） | 需检测异常值（如 <= 0 或 > 500） |
| `is_canceled` | 目标标签 | **严禁删除**，作为 y 使用 |
| `reservation_status` | 最终状态（Check-Out/Canceled） | **永久物理删除** |
| `reservation_status_date` | 最终状态日期 | **永久物理删除** |

## 第三章：人类决策清单（AI 必须等待人工确认的节点）

在以下关键节点，AI 必须先输出方案，等待人类输入"确认"或"按此执行"后，才能继续编码。这些决策点构成了答辩核心素材。

### 决策点 D-01（特征公式）
合成特征 `total_stay` 和 `total_guests` 的计算公式，由人类决定。

### 决策点 D-02（算法选型）
选择 **XGBoost** 还是随机森林，由人类决定（**人类选定：XGBoost**，因其处理表格数据更优）。

### 决策点 D-03（异常值阈值）
`adr` 的上限阈值（**人类定为 500**）和下限阈值（**人类定为 0**）。

### 决策点 D-04（图表默认视图）
看板默认展示全部数据还是采样数据（**人类定为采样 5000 条快速展示**，点击按钮加载全量）。

### 决策点 D-05（预测阈值）
分类阈值默认设为 0.5，还是针对酒店业务调整（**人类定为 0.5**，但提供阈值滑块让用户手动调节）。

## 第四章：分步执行计划（AI 严格按此顺序工作）

> **AI 注意**：每一大步完成后，**必须暂停**，输出结果摘要，等待人类发出"继续下一步"的指令。**不得一次性生成全部代码。**

### 第 0 步：环境确认与数据加载（探索性）

- **任务**：读取 CSV，输出前 5 行、数据形状、各列缺失值比例
- **禁止**：不要做任何数据清洗或删除操作，只做"只读"观察
- **产出**：控制台打印数据摘要报告

### 第 1 步：数据清洗与物理删除（铁腕手段）

- **任务**：
  1. 立即删除 `reservation_status` 和 `reservation_status_date` 两列（直接 drop，无任何商量余地）
  2. 处理缺失值：`children` 空值填 0，`agent` 空值填 0，`country` 空值填 'Unknown'
  3. 剔除 `adr` <= 0 或 > 500 的异常记录
- **产出**：清洗后的新数据集形状（行数应略少于 12 万），并保存为 `data/processed/cleaned_data.csv`

### 第 2 步：人类决策驱动的特征工程

- **任务**：严格按照以下人类指定的公式创建新特征（**不得增加额外特征**）：

  ```python
  total_stay = stays_in_weekend_nights + stays_in_week_nights
  total_guests = adults + children + babies
  arrival_month_num: 将 arrival_date_month（如 'July'）映射为数字（7）
  ```

- **产出**：包含新特征的数据集，并输出新特征的统计描述

### 第 3 步：划分数据集（严防数据泄露）

- **任务**：
  1. 定义 **X（特征）**：删除 `is_canceled` 列之外的所有列
  2. 定义 **y（标签）**：单独取出 `is_canceled` 列
  3. 按 **80% 训练 / 20% 测试** 划分，使用 `stratify=y` 保证类别平衡
  4. 对数值特征进行 `StandardScaler` 标准化（用训练集拟合，转换训练集和测试集）
- **产出**：`X_train`、`X_test`、`y_train`、`y_test`

### 第 4 步：静态与交互式可视化看板开发

- **任务**：使用 Plotly 开发 Web 看板，展示以下内容（分为静态 Tab 和交互 Tab）：
  - 总览指标：总订单数、取消率、平均房价、平均提前预订天数（KPI 卡片）
  - 取消率对比：按酒店类型（Resort vs City）的取消率柱状图
  - 时间趋势：按月聚合的取消率趋势折线图（带时间范围滑块）
  - 特征分布：`lead_time` 和 `adr` 的直方图（交互式可缩放）
  - 相关性热力图：展示各特征与 `is_canceled` 的相关性
- **产出**：可运行的 HTML 页面或 Flask 路由代码

### 第 5 步：XGBoost 模型训练（核心建模）

- **任务**：
  1. 使用 **XGBoost 分类器**
  2. 超参数由人类指定（直接使用）：
     ```python
     n_estimators=200, max_depth=6, learning_rate=0.1, tree_method='hist'
     ```
  3. 使用 `X_train`、`y_train` 训练模型
  4. 在 `X_test`、`y_test` 上评估，输出：**混淆矩阵**、**分类报告**（精确率/召回率/F1）、**ROC-AUC 值**
- **产出**：模型评估报告，并保存模型为 `models/xgboost_model.pkl`

### 第 6 步：数据泄露危害量化对照实验（答辩高光环节）

- **任务**：
  1. 加载原始的、未被清洗的旧数据集（包含 `reservation_status` 列）
  2. 用第 5 步训练好的 `xgboost_model.pkl` 去预测旧数据集（只选取 `X_train` 中有的列名）
  3. 计算在旧数据集上的"虚假准确率"
  4. 输出对比表格：

| 评估对象 | 准确率 | AUC | 说明 |
|----------|--------|-----|------|
| 干净测试集 | （真实值，如 0.82） | （真实值） | 模型真实泛化能力 |
| 含泄露列的旧数据 | （虚假值，如 0.99） | （虚假值） | 数据泄露导致的虚高 |

- **产出**：对比结论——"数据泄露导致模型性能虚高了约 XX%"

### 第 7 步：简单 Web 应用整合（Flask）

- **任务**：将上述所有功能整合为一个 Flask Web 应用，包含：
  - 首页（项目简介）
  - 数据看板（展示第 4 步的图表）
  - 模型评估页（展示第 5 步的评估指标和特征重要性图）
  - 批量预测页（用户可上传新 CSV，下载带预测结果的 CSV）
- **产出**：完整的 `app.py` 和前端模板文件
